# 任务：实现 MarkdownReader Word 式多窗口

## 背景与目标

将 MarkdownReader 从单窗口架构升级为 Word 式多窗口：每个窗口拥有独立的文件、目录、编辑、查找、大纲、面板和 Undo 状态；应用级能力保持共享且只执行一次；所有菜单命令、外部打开和资源所有权均能准确路由到目标窗口。

## 权威输入文档

实现、评审和验收以以下三个文件为准：

1. 产品需求：`docs/multi-window-requirements.md`
2. 技术设计：`docs/plans/2026-07-14-multi-window-design.md`
3. 实施拆解：`docs/plans/2026-07-14-multi-window-implementation-plan.md`

文档优先级为“产品需求 > 技术设计 > 实施拆解”。如果实现中发现文档冲突或现有代码约束导致方案不可行，应先记录差异并调整设计或计划，不得静默改变已确认的产品行为。

## 工作范围

- 完成需求文档中的 `MW-01` 至 `MW-15`。
- 按实施拆解中的 Task 1 至 Task 14 顺序推进，并遵循其中的测试、迁移和检查点要求。
- 建立窗口级 `WindowSession`、应用级 `WindowCoordinator`、资源所有权及打开路由机制。
- 将窗口级菜单迁移到基于焦点窗口的命令路由，隔离 Undo、面板、拖拽、关闭确认和窗口 UI 状态。
- 统一 Finder、Open Recent、OpenPanel、命令行及应用内导航等文件打开入口。
- 补充自动化测试，完成双窗口/多窗口人工回归，并同步架构、需求、CHANGELOG 和发布说明。

## 不可变更的行为边界

- 采用 Word 式多窗口，不支持同一 Markdown 文件同时存在于多个窗口。
- 同一文件只允许一个所有者窗口；再次打开时激活已有窗口，不创建重复文档实例。
- 若一个窗口已打开 `A.md`，另一个目录窗口点击目录树中的 `A.md`，目录窗口保持当前有效选择和文档不变，并激活所有者窗口；该行显示“已在另一窗口打开”。
- 窗口级命令只作用于当前焦点窗口，不允许继续依赖无目标的全局通知广播。
- 更新检查、WebView 预热等应用级服务只能幂等执行一次。
- 不得以全局 `nonisolated(unsafe)` 可变状态规避 Swift 6 并发隔离。

## 执行要求

- 使用独立功能分支或 worktree 实施，保留工作区中与本任务无关的用户改动。
- 按 TDD 推进核心模型和路由：先写失败测试，再实现最小代码并验证通过。
- 在新的窗口路由、生命周期和关闭保护具备等价能力前，不得提前删除现有单窗口保护逻辑。
- 所有用户可见文案通过 `L10n` 提供简中、繁中和英文；文件操作继续通过 `FileService`。
- 每一阶段均按实施拆解执行局部测试与构建；发现架构偏差时同步更新对应文档。

## 不包含

除需求文档“非目标”章节外，本任务明确不包含：同一文件多窗口编辑、标签页/分屏平铺、完整窗口会话恢复、云同步与协作，以及迁移到 `DocumentGroup`。

## 验收标准

- 需求文档中 `MW-01` 至 `MW-15`、P0/P1 验收项及测试矩阵全部通过。
- 实施拆解中的 Task 1 至 Task 14 均完成，旧的单窗口广播、全局 active undo 和失效保护机制已按计划清理。
- 多窗口下打开、切换、保存、Undo/Redo、关闭、退出、Dock 重开、拖拽、PDF 导出和外部多 URL 打开行为符合设计文档。
- `A.md` 跨窗口所有权冲突场景与已确认行为一致，且不存在选择闪烁、错误加载或重复所有权。
- 应用级服务无重复执行，窗口关闭后无残留注册、观察者或明显内存泄漏。
- 相关架构文档、需求追踪、CHANGELOG 和发布说明已同步。
